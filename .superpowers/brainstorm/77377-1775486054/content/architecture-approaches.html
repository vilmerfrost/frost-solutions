Architecture Approach

How should binders, checklists, and cases relate in the database?

A

Flat Tables — Keep It Simple

Separate tables for binders, tabs, documents, checklists, checklist_items, cases. Linked by foreign keys. Similar to the current project_documents structure but with binder/tab layers added on top.

Pros

  • Simple to query and understand
  • Easy RLS policies (tenant_id on every table)
  • Fits your existing Supabase patterns
  • Fast to implement

Cons

  • Many JOINs for nested views (binder → tab → documents)
  • Template system needs separate template tables that mirror the structure
  • More migrations as features grow
B

JSONB Hybrid — Flexible Structure

Binders store their tab structure as JSONB (name, order, icon, permissions). Documents, checklists, and cases remain as relational tables referencing binder_id + tab_key. Templates are just JSONB snapshots.

Pros

  • Tab structure is flexible without migrations
  • Templates = copy the JSONB structure (trivial)
  • Fewer tables overall
  • Tab reordering/renaming is a single update

Cons

  • Can't enforce FK constraints on tab_key
  • JSONB querying is less intuitive
  • Slightly harder to set per-tab RLS
C

Recommended: Relational Core + JSONB Config

Binders and tabs are relational tables (proper FKs, RLS). But each tab has a JSONB config column for flexible metadata (icon, color, custom fields). Templates stored as a separate binder_templates table with a JSONB structure column that snapshots the tab layout.

Pros

  • Best of both: relational integrity + flexibility
  • RLS works naturally on binder/tab level
  • Templates are clean JSONB snapshots
  • Easy to extend without schema changes
  • Matches your existing patterns (relational + config)

Cons

  • Slightly more tables than option B
  • Need to keep JSONB config and relational data in sync
